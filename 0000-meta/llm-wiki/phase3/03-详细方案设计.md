### 三层职责收束

| 层级 | 定位 | 目录 | 生命周期 |
|------|------|------|----------|
| L1 原始数据层 | 未加工原始素材 | `0003-inbox/` | 临时层，完成 L2 标准化后可删除或归档 |
| L2 主题知识层 | L1 的标准化数据，按主题目录存放 | `{编号}-{主题名}/` | 主数据层，保持来源约束和结构统一 |

- 写入逻辑：L1 → L2 → L3
- 查询逻辑：L3 → L2 → L1（L1 仅在需要追溯原始材料且文件仍保留时使用）

## 3.2 Schema 与元数据标准 ✅

### Frontmatter 完整定义

```yaml
---
title: "Transformer"           # 知识标题
topic: "3001-深度学习"         # 所属主题目录名，Claude 自动填写
layer: L2                      # L2 主题知识层
kind: standard                 # L2 固定为 standard；L3 使用 topic/concept/entity/comparison
tags: [NLP, 注意力机制, LLM]    # 8-10 个关键词
aliases: [transformer, 变换器, 自注意力架构]  # 含 slug 的别名列表
created: 2026-05-18            # 创建日期
updated: 2026-05-18            # 更新日期
source: manual                 # manual | url | file | claude
status: draft                  # draft | published
summary: "基于自注意力机制的序列建模架构"
---
```

### 字段详细定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `title` | string | ✅ | 知识标题，用作 Obsidian 页面名 |
| `topic` | string | ✅ | 所属主题目录名，Claude 自动填写，Base 分组主维度 |
| `layer` | enum | ✅ | L1 / L2 / L3；主题知识层为 L2 |
| `kind` | enum | ✅ | L2 固定为 standard；L3 为 topic / concept / entity / comparison |
| `tags` | list | | 8-10 个关键词 |
| `aliases` | list | | 含 slug 的别名列表，Obsidian 搜索和链接补全使用 |
| `created` | date | ✅ | YYYY-MM-DD |
| `updated` | date | ✅ | YYYY-MM-DD，每次修改时更新 |
| `source` | enum | | manual / url / file / claude |
| `status` | enum | ✅ | draft / published |
| `summary` | string | ✅ | 一句话摘要，≤50字 |
| `processing_path` | string | | L3 专用，加工路径：`{大类}/{主题域}` |
| `entry_count` | number | | L3 专用，当前页引用的 L2 条目数量 |
| `source_topics` | list | | L3 专用，覆盖的 L2 主题目录列表 |
| `source_url` | string | | L1 专用，原始 URL（仅 inbox） |

> **分层说明**：`created` 为 L2 必填，L3 以 `updated` 代创建时间；`status` L1 用 `raw/processing/archived`，L2/L3 用 `draft/published`；`processing_path`/`entry_count`/`source_topics` 仅 L3 使用；`source_url` 仅 L1 inbox 使用。Schema 校验时需按 `layer` 区分字段要求。

### status 流转

```
draft ──→ published
  │
  └── 重新编辑（回到 draft）
```

- `draft`：工作中，Claude Ingest 时默认写入
- `published`：可正式引用，用户手动确认

## 3.3 数据流转全流程 ✅

### 五操作详述

| 操作 | 触发 | 输入 | 输出 | 副作用 |
|------|------|------|------|--------|
| **Ingest** | 用户提供 + Claude | inbox 文件/URL/文本 | 新知识条目 (draft) | LOG、可能新建主题目录、可能更新 0101-0104 |
| **Update** | 用户要求 | 已有文件路径或 wikilink | 更新后条目 | 更新 updated、LOG、重大变更时级联 L3 |
| **Publish** | 人工确认 | draft 条目 | status→published | — |
| **Query** | 用户提问 | 自然语言 | 基于 L3→L2→L1 的回答 + 引用 | 可选存档 outbox |
| **Lint** | 用户触发 | — | 问题报告 | 自动修复项直接执行 |

### Ingest 步骤

```
1. 素材放入 0003-inbox/（L1 原始数据层）
2. 用户 Claude Code: @"0003-inbox/xxx.md" 处理
3. Claude 读取 → 判断归属主题目录 → 不存在则直接新建并告知用户（3.9）
4. 根据标题生成 slug，并用 `rg --files` 检查目标主题目录内是否重名；重名时追加 `-2/-3`
5. 按 `t-knowledge.md` 创建 L2 标准化主题知识条目 `{slug}.md`
6. 判断是否需要新建/更新 L3 加工页面（3.7），超过50条自动分页
7. 追加 0109-log/LOG-YYYY-MM-DD.md
8. 报告：新建 `{主题目录}/{slug}.md`，建议关联
9. 询问用户是否删除 inbox 原文件，确认后删除
```

### Query 步骤

```
1. 用户提问
2. 优先检索 L3：`0101-0104/` 中的 title/processing_path/summary/正文，寻找已加工知识
3. L3 证据不足时回查 L2：检索主题目录中的标准化知识条目
4. L2 仍不足且 L1 文件仍保留时，最后回查 `0003-inbox/` 原始素材
5. 默认读 Top 8 全文；证据不足时 Claude 可扩大范围，但需在回答中说明当前检索层级
6. 整理回答，引用 [[wikilink]]
7. 用户可选："存档到 outbox"
```

### Update 步骤（三级变更）

```
1. 用户指定文件路径、标题或 wikilink，描述变更
2. Claude 读取 → 判断变更等级 → 执行对应操作：

   轻微变更（正文措辞/错别字）：
     → 修改正文 → 更新 updated → 追加 LOG

   中等变更（tags/aliases）：
     → 修改 Frontmatter → 追加 LOG

   重大变更（title/summary/核心事实）：
     → 修改正文/Frontmatter
     → Grep 搜索所有 [[wikilink]] 引用本知识的文件
     → 报告受影响文件，用户确认后更新引用
     → 同步更新 0101-0104 加工页面
     → 追加 LOG

3. git commit
```

### Lint 检查清单

**自动修复**：
- INDEX.md 与实际主题目录不一致 → 更新
- 内部 [[wikilink]] 指向不存在 → 标记

**报告（需人工）**：
- 孤立页面（无任何 incoming wikilink，Glob vs Grep 比对）
- 长期 draft >30 天
- 缺失跨主题引用建议
- Frontmatter 不完整
- summary 超长或缺失

## 3.7 0101-0104 知识加工层维护规范

### 定位

0101-0104 是 Claude 主动维护的知识加工层，内容基于 L2 主题知识层进行拆解、重构、聚合、扩写。它不是原始数据层，也不是简单视图层。

### 触发时机

| 触发 | 判断标准 | Claude 动作 |
|------|----------|-----------|
| Ingest 新知识 | 该知识是否属于已有主题域？ | 是 → 追加条目引用到已有加工页面；否 → 新建主题域（见下） |
| Ingest 触发新主题域 | 目标知识无法归入任何已有主题域 | 在 0101/0102/0103/0104 四个目录下建对应大类目录和主题域 `.md` |
| Update 已有知识 | summary/核心事实是否变更？ | 是 → 更新加工页面中的对应引用；否 → 不触发 |

**归入判断规则**：Claude 对比新知识的 `topic`（如 `3001-深度学习`）与已有 L3 加工页面的 `processing_path`，匹配则归入。无匹配时触发新建。

### 大类↔主题目录映射

| 大类名 | 对应千位 | 匹配规则 | 示例主题目录 |
|--------|----------|----------|-------------|
| 个人生活 | 1-2 | topic 以 1xxx 或 2xxx 开头 | 1101-健康、1201-投资理财 |
| AI技术 | 3 | topic 以 3xxx 开头 | 3001-深度学习、3002-LLM |
| 工作业务 | 4 | topic 以 4xxx 开头 | 4001-项目管理 |
| 通用基础 | 5+ | topic 以 5xxx+ 开头 | 5001-计算机科学、5101-物理学 |

> 映射是约定规则，无需显式配置文件。Claude 从主题目录编号千位推导大类名。

### 0105-wiki-base 定位

Base 面板做**跨层浏览**入口，按 `topic`、`layer`、`kind` 和 `tags` 过滤。P0 先提供标准 `.base` 模板；用户可通过 Obsidian Base GUI 做个性化视图，Claude 默认只维护标准模板，不改用户自定义视图。

```
0105-wiki-base/
├── 全部知识.base                 → 所有主题目录
├── Draft待发布.base              → status = draft
├── 最近更新.base                 → 按 updated 倒序
├── 按层级.base                   → group by layer
├── AI技术-全部.base              → topic 千位 = 3
├── 工作业务-全部.base             → topic 千位 = 4
├── 按内容形态.base               → group by kind
└── 按标签.base                   → group/filter by tags
```

### L3 加工页面格式

```markdown
---
title: "{主题域}概念索引"
layer: L3
kind: concept
processing_path: "{大类}/{主题域}"
updated: "YYYY-MM-DD"
entry_count: 48
source_topics: [{{source_topics}}]
---

# {主题域}

{主题域说明，1-2句话}

## 条目列表
- [[{slug}]] — {summary}
```

### 规模管理

单页面条目上限 **50 条**。Claude Ingest 时检查当前页条目数，满则自动创建分页：
```
AI技术/深度学习.md       (满50条)
AI技术/深度学习-2.md     (自动)
```
Update 时只更新对应分页。用户查看无感知。

## 3.8 L1 原始素材生命周期

### 三态

```
raw（待处理）→ processing（处理中）→ 已标准化到 L2 → 归档/删除
```

### Ingest 完成后

1. Claude 报告 L1→L2 标准化结果
2. L2 已校验通过后，L1 原始文件不再作为主数据源
3. 询问用户："是否删除 inbox 原文件？"
4. 用户确认 → 删除原文件
5. 用户拒绝 → 文件保留，status 标记 `archived`，仅作为追溯材料

## 3.9 主题目录创建规范

Ingest 时若知识无归属，Claude 直接创建：

1. Glob 当前所有主题目录 → 确认无匹配目录
2. 确定编号：同千位已有目录取最大编号 +1
3. 直接创建目录，在 `0100-wiki-meta/INDEX.md` 中登记
4. 报告中告知用户：已新建主题目录 `{编号}-{主题名}`
5. 用户不满意可要求修改

**命名规则**：
- 格式：`{4位编号}-{中文主题名}`
- 编号 = Glob 同千位已有目录取最大编号 +1
- 示例：已有 `3001-深度学习`，新建 AI 技术主题 → `3002-{主题名}`

## 3.10 操作收尾与失败恢复（待落地）
