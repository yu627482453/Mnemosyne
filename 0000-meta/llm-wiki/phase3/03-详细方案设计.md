## 3.2 Schema 与元数据标准 ✅

### Frontmatter 完整定义

```yaml
---
id: "3001-00001"              # {目录编号}-{5位序号}
title: "Transformer"           # 知识标题
topic: "3001-深度学习"         # 所属主题目录名，Claude 自动填写
tags: [NLP, 注意力机制, LLM]    # 8-10 个关键词
aliases: [transformer, 变换器, 自注意力架构]  # 含 slug 的别名列表
created: 2026-05-18            # 创建日期
updated: 2026-05-18            # 更新日期
source: manual                 # manual | url | file | claude
status: draft                  # draft | published
summary: "基于自注意力机制的序列建模架构"
---
```

### 字段详细定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | ✅ | `{目录编号}-{5位序号}`，唯一标识 |
| `title` | string | ✅ | 知识标题，用作 Obsidian 页面名 |
| `topic` | string | ✅ | 所属主题目录名，Claude 自动填写，Base 分组主维度 |
| `tags` | list | | 8-10 个关键词 |
| `aliases` | list | | 含 slug 的别名列表，Obsidian 搜索和链接补全使用 |
| `created` | date | ✅ | YYYY-MM-DD |
| `updated` | date | ✅ | YYYY-MM-DD，每次修改时更新 |
| `source` | enum | | manual / url / file / claude |
| `status` | enum | ✅ | draft / published |
| `summary` | string | ✅ | 一句话摘要，≤50字 |

### status 流转

```
draft ──→ published
  │
  └── 重新编辑（回到 draft）
```

- `draft`：工作中，Claude Ingest 时默认写入
- `published`：可正式引用，用户手动确认

### 模板选择指引

| 模板 | 适用特征 | 目标目录 |
|------|----------|----------|
| `t-concept.md` | 需要定义+解释的抽象概念、术语、理论 | 主题目录 |
| `t-entity.md` | 有固定属性可列表的具体对象（人/组织/产品/工具） | 主题目录 |
| `t-topic.md` | 跨概念的综合论述、专题综述 | 主题目录 |
| `t-comparison.md` | 两个或多个对象的对比分析 | 主题目录 |
| `t-inbox.md` | 原始素材 | `0003-inbox/` |
| `t-daily.md` | 日记 | 日记目录 |

## 3.3 数据流转全流程 ✅

### 五操作详述

| 操作 | 触发 | 输入 | 输出 | 副作用 |
|------|------|------|------|--------|
| **Ingest** | 用户提供 + Claude | inbox 文件/URL/文本 | 新知识条目 (draft) | LOG、可能新建主题目录、可能更新 0101-0104 |
| **Update** | 用户要求 | 已有知识 ID | 更新后条目 | 更新 updated、LOG、重大变更时级联合成层 |
| **Publish** | 人工确认 | draft 条目 | status→published | — |
| **Query** | 用户提问 | 自然语言 | 合成回答 + 引用 | 可选存档 outbox |
| **Lint** | 用户触发 | — | 问题报告 | 自动修复项直接执行 |

### Ingest 步骤

```
1. 素材放入 0003-inbox/
2. 用户 Claude Code: @"0003-inbox/xxx.md" 处理
3. Claude 读取 → 判断归属主题目录 → 不存在则直接新建并告知用户（3.9）
4. Bash: ls 目标目录取最大序号 +1（空目录从 00001 开始）
5. 按模板创建 {编号}-{序号}-{slug}.md
6. 判断是否需要新建/更新 0101-0104 合成页面（3.7），超过50条自动分页
7. 追加 0109-log/LOG-YYYY-MM-DD.md
8. 报告：新建 {id}，建议关联
9. 询问用户是否删除 inbox 原文件，确认后删除
```

### Query 步骤

```
1. 用户提问
2. Claude Grep title/summary 字段 → 定位候选文件（不经过中间索引）
3. 按匹配度排序，读 Top 8 全文（软阈值，Claude 可自主决定读更多）
4. 合成回答，引用 [[wikilink]]
5. 用户可选："存档到 outbox"
```

### Update 步骤（三级变更）

```
1. 用户指定知识 ID，描述变更
2. Claude 读取 → 判断变更等级 → 执行对应操作：

   轻微变更（正文措辞/错别字）：
     → 修改正文 → 更新 updated → 追加 LOG

   中等变更（tags/aliases）：
     → 修改 Frontmatter → 追加 LOG

   重大变更（title/summary/核心事实）：
     → 修改正文/Frontmatter
     → Grep 搜索所有 [[wikilink]] 引用本知识的文件
     → 报告受影响文件，用户确认后更新引用
     → 同步更新 0101-0104 合成页面
     → 追加 LOG

3. git commit
```
```

### Lint 检查清单

**自动修复**：
- INDEX.md 与实际主题目录不一致 → 更新
- 内部 [[wikilink]] 指向不存在 → 标记

**报告（需人工）**：
- 孤立页面（无任何 incoming wikilink，Glob vs Grep 比对）
- 长期 draft >30 天
- 缺失跨主题引用建议
- Frontmatter 不完整
- summary 超长或缺失

## 3.4 Git 版本控制策略 ✅

### 分支策略

日常直接在 master 提交（单人使用，trunk-based）。批量操作可选开分支。

### 提交方式

全部手动提交。Claude 操作后直接执行 git 命令。

| 场景 | 格式 |
|------|------|
| Claude 知识操作 | `wiki: {操作} {id} — {摘要}` |

### .gitignore

```gitignore

*.tmp
*.bak
.DS_Store

```

### Claude 提交流程

```
git status → git add {文件} → git commit -m "wiki: ..." → git push

```

### 提交粒度

| 操作 | 变更文件 | 提交 |

## 3.7 0101-0104 合成层维护规范

### 定位

0101-0104 是 Claude 主动写作的合成知识层，内容从主题目录的知识原子提取、重组、扩写。不是视图层。

### 触发时机

| 触发 | 判断标准 | Claude 动作 |
|------|----------|-----------|
| Ingest 新知识 | 该知识是否属于已有主题域？ | 是 → 追加条目引用到已有合成页面；否 → 新建主题域（见下） |
| Ingest 触发新主题域 | 目标知识无法归入任何已有主题域 | 在 0101/0102/0103/0104 四个目录下建对应大类目录和主题域 `.md` |
| Update 已有知识 | summary/核心事实是否变更？ | 是 → 更新合成页面中的对应引用；否 → 不触发 |

**归入判断规则**：Claude 对比新知识的 `topic`（如 `3001-深度学习`）与已有合成页面的 `domain_path`，匹配则归入。无匹配时触发新建。

### 大类↔主题目录映射

| 大类名 | 对应千位 | 匹配规则 | 示例主题目录 |
|--------|----------|----------|-------------|
| 个人生活 | 1-2 | topic 以 1xxx 或 2xxx 开头 | 1101-健康、1201-投资理财 |
| AI技术 | 3 | topic 以 3xxx 开头 | 3001-深度学习、3002-LLM |
| 工作业务 | 4 | topic 以 4xxx 开头 | 4001-项目管理 |
| 通用基础 | 5+ | topic 以 5xxx+ 开头 | 5001-计算机科学、5101-物理学 |

> 映射是约定规则，无需显式配置文件。Claude 从 ID 千位推导大类名。

### 0105-wiki-base 定位

Base 面板做**跨主题浏览**入口，按 `topic` 和 `tags` 过滤。用户通过 Obsidian Base GUI 创建 `.base` 视图文件，Claude 不参与其维护。

### 合成页面格式

```markdown
---
title: "{主题域}概念索引"

updated: "YYYY-MM-DD"
entry_count: 48
---

# {主题域}

{主题域说明，1-2句话}

## 条目列表
- [[{id}-{slug}]] — {summary}
```

### 规模管理

单页面条目上限 **50 条**。Claude Ingest 时检查当前页条目数，满则自动创建分页：
```
AI技术/深度学习.md       (满50条)
AI技术/深度学习-2.md     (自动)
```
Update 时只更新对应分页。用户查看无感知。

## 3.8 inbox 素材生命周期

### 三态

```
raw（待处理）→ processing（处理中）→ 归档/删除
```

### Ingest 完成后

1. Claude 报告结果
2. 询问用户："是否删除 inbox 原文件？"
3. 用户确认 → 删除原文件
4. 用户拒绝 → 文件保留，status 标记 `archived`

## 3.9 主题目录创建规范

Ingest 时若知识无归属，Claude 直接创建：

1. Glob 当前所有主题目录 → 确认无匹配目录
2. 确定编号：同千位已有目录取最大编号 +1
3. 直接创建目录，在 `0100-wiki-meta/INDEX.md` 中登记
4. 在 `0100-wiki-meta/INDEX.md` 中登记
5. 报告中告知用户：已新建主题目录 `{编号}-{主题名}`
6. 用户不满意可要求修改

**命名规则**：
- 格式：`{4位编号}-{中文主题名}`
- 编号 = Glob 同千位已有目录取最大编号 +1
- 示例：已有 `3001-深度学习`，新建 AI 技术主题 → `3002-{主题名}`
