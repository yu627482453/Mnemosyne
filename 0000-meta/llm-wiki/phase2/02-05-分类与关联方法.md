# 2.5 知识分类与关联方法

> 日期：2026-05-14

## 一、分类体系设计

### 双轴分类模型

知识条目沿两个维度同时分类：

- **Domain（领域）** = 主题目录编号的千位：千位 1-2 个人生活 / 3 AI技术 / 4 工作业务 / 5+ 通用基础
- **Type（类型）** = Frontmatter `type` 字段：`concept` / `entity` / `topic` / `comparison`

```
           TYPE (类型)
           concept    entity    topic    comparison
DOMAIN     ────────────────────────────────────────
千位 1-2   0101       0102      0103      0104      (个人生活 MOC)
千位 3     0101       0102      0103      0104      (AI技术 MOC)
千位 4     0101       0102      0103      0104      (工作业务 MOC)
千位 5+    0101       0102      0103      0104      (通用基础 MOC)
```

### 各分类方式的角色

| 方式 | 角色 | 实现 |
|------|------|------|
| **主题目录** | 物理存放位置，一个主题一个顶级目录 | `{编号}-{主题名}/`，如 `3001-深度学习/` |
| **Frontmatter domain** | 所属主题目录名，Base 分组维度 | `domain: "3001-深度学习"` |
| **Frontmatter type** | 知识类型标注 | `type: concept\|entity\|topic\|comparison` |
| **ID 前缀** | 编号即目录号，千位隐含领域 | `3001-00001` → AI技术 |
| **Frontmatter tags** | 灵活多维标签（P3 重点建设） | `tags: [tag1, tag2]` |
| **Base 视图** | GUI 导航，按 type/tag 动态过滤 | `0105-wiki-base/*.base` |
| **Dataview 查询** | 复杂聚合 | `0100-wiki-meta/INDEX.md` |

### 主分类方案：主题目录 + Type Overlay

**知识文件存放规则**：
- 文件存放在对应主题目录下，目录内平铺
- Type 通过 Frontmatter 标注
- 文件命名：`{编号}-{5位序号}-{slug}.md`，如 `3001-00001-transformer.md`

```
3001-深度学习/
├── _index.md
├── _counter.md
├── 3001-00001-transformer.md      type: concept
├── 3001-00002-cnn.md             type: concept
├── 3001-00003-cnn-vs-transformer.md  type: comparison
└── ...
```

**0101-0104 关系层的定位**：
- 不是存放知识文件本身，而是存放 **MOC 索引页** 和 **跨域综合页面**
- 通过 Dataview/Base 从主题目录动态聚合同类型知识
- 例：`0101-wiki-concepts/AI概念索引.md` 聚合所有千位 3 的 `type: concept` 条目

### 三层数据流中的分类位置

```
0003-inbox/                    原始素材，未分类
       ↓
{编号}-{主题名}/               数据基础层，按主题存放，标注 type
       ↓
0101-0104/                     数据关系层，MOC + 跨域综合
       ↓
0105-wiki-base/                Base 导航面板，GUI 浏览入口
```

## 二、关联策略设计

### 四种关联方式对比

| 方式 | 性质 | 优势 | 劣势 |
|------|------|------|------|
| **[[wikilink]]** | 持久化双向链接 | Obsidian 原生，Graph 可视化，反向链接自动追踪 | 需手动维护 |
| **MOC 索引页** | 人工策划的大纲 | 结构化导航，可加注释，适合"入口" | 需手动更新 |
| **Dataview 查询** | 动态自动聚合 | 自动发现新知识，无需维护 | 学习成本，性能随量增长 |
| **Canvas 连线** | 视觉探索 | 空间化思维，临时梳理 | 连线不生成 wikilink，非持久化 |
| **Base 视图** | GUI 过滤/分组 | 零代码，即时切换维度 | 不如 Dataview 灵活 |

### 组合方案

```
持久化关联：[[wikilink]]
    知识页面间互相引用，构建长久的链接网络
    ↓
结构化导航：MOC + Base
    MOC 做主题入口，Base 做按 type/domain/tag 的动态过滤
    ↓
动态发现：Dataview
    INDEX.md 全局聚合，自动发现新知识和孤立页面
    ↓
视觉探索：Canvas
    专题梳理、头脑风暴、关系总览
```

### 关联的层级

| 层级 | 方式 | 示例 |
|------|------|------|
| **页面内** | `[[wikilink]]` | `transformer 中使用了 [[自注意力]] 机制` |
| **页面底** | `links` Frontmatter | `links: [3000-00001, 5000-00005]` |
| **跨域索引** | Dataview in INDEX.md | 列出所有 type=concept 的页面 |
| **入口导航** | Base in 0105 | 按域/类型/标签过滤浏览 |
| **关系图谱** | Graph + Canvas | 可视化链接网络 |

## 三、具体实现规则

### 3.1 知识创建时的分类决策

Claude 处理 inbox 时按以下规则决定分类：

1. 确定 **Domain**：内容属于个人生活/AI技术/工作业务/通用基础？
2. 确定 **Type**：这是概念定义/实体描述/综合主题/对比分析？
3. 确定 **ID**：在 Domain 下取最大序号 +1
4. 确定 **Tags**：提取 2-5 个关键词标签
5. 识别 **Links**：从内容中提取到已有知识页面的引用

### 3.2 wikilink 使用规范

```markdown
# 推荐：使用页面标题
transformer 中的 [[自注意力机制]] 是核心创新

# 不推荐：使用路径
transformer 中的 [[5000-00002-self-attention|自注意力]] 
# (路径仅在所有 wikilink 相同标题时用于消歧)
```

### 3.3 MOC 页面结构

```markdown
# AI 技术概念索引

## 基础架构
- [[transformer]]
- [[自注意力机制]]
- [[位置编码]]

## 训练方法
- [[预训练]]
- [[微调]]
- [[RLHF]]
```

## 四、2.5 结论

| 维度 | 选择 | 说明 |
|------|------|------|
| **主分类** | Domain-Primary（目录）+ Type Overlay（Frontmatter） | 双轴分类，Domain 定物理位置，Type 定逻辑类型 |
| **0101-0104 定位** | MOC 索引 + 跨域综合页 | 不放知识文件本身，靠 Dataview/Base 动态聚合 |
| **持久化关联** | `[[wikilink]]` | Obsidian 原生，全系统联动 |
| **结构化导航** | Base（GUI 浏览）+ MOC（入口页） | Base 做日常，MOC 做深度 |
| **动态聚合** | Dataview（INDEX.md） | 全局索引，自动发现 |
| **视觉探索** | Canvas + Graph | 非持久化探索，临时梳理 |
| **Tags（P3）** | Frontmatter 预留，未来建设层级标签体系 | 本次不实现 |

**关键判断**：不把所有知识文件平铺到 0101-0104，而是用 **Domain 目录 + Type Frontmatter + Base 视图** 实现多维分类，避免同一知识在多个目录下出现副本。
