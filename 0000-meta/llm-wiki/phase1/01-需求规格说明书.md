## 一、项目概述

基于 Obsidian + Claude Code 构建的智能本地 Wiki 系统。以 Obsidian 为知识前端载体，Claude Code 为后端智能引擎，实现知识从摄入、加工、关联到检索的全流程管理。

### 核心设计思想

1. **渐进式知识披露**：通过结构化目录和 Base 面板实现知识分层导航
2. **LLM 知识管家**：Claude 作为知识管理者，负责加工、关联、维护和问答
3. **层级前缀 ID 体系**：格式 `{层级}-{序号}`，实现精确检索和关系管理
4. **三层数据架构**：原始数据 → 数据基础层 → 数据关系层，渐进式加工
5. **Git 原生支持**：全系统版本控制，追踪知识变更历史

## 二、MVP 分阶段定义

### 阶段 A：最小 MVP（P0 — 立即实现）

**目标**：建立可用的基础 Wiki 框架，支持手动知识管理

| 功能 | 描述 |
|------|------|
| 目录结构搭建 | 按设计方案创建完整目录树 |
| 知识模板系统 | 基于 Templater 创建各类型笔记模板 |
| 手动笔记创建 | 用户通过 Obsidian 手动创建和编辑知识条目 |
| Git 版本控制 | 手动 git 提交，Claude 操作后直接执行 git commit + push |
| 基础元数据规范 | 定义并落实 frontmatter 元数据标准 |
| Wiki 元数据骨架 | 创建 0100-wiki-meta/（INDEX.md/SCHEMA.md/RELATIONS.md）和 0109-log/ 基础结构 |

### 阶段 B：中等 MVP（P1 — 最小 MVP 完成后）

**目标**：引入 Claude Code 辅助知识加工

| 功能 | 描述 |
|------|------|
| Claude Code 工作流脚本 | 编写 Claude 处理笔记的提示词和脚本 |
| Inbox 处理流程 | 用户将原始知识放入 inbox，手动触发 Claude 加工 |
| 知识自动分类 | Claude 分析内容，建议所属层级和分类 |
| 摘要与标签生成 | Claude 自动生成摘要、标签和元数据 |
| 知识链接建议 | Claude 识别知识间潜在关联并建议双向链接 |
| Outbox 输出区 | Claude 生成的报告、总结等输出至 outbox |

### 阶段 C：完整 MVP（P2 — 中等 MVP 完成后）

**目标**：知识关联图谱和智能检索

| 功能 | 描述 |
|------|------|
| 知识关系图谱 | 基于 Canvas + Graph 构建可视化关联图，wikilink 做持久化关联 |
| 自然语言检索 | 通过 Claude 实现自然语言知识问答 |
| 全自动入库 | Claude 从 URL/文件自动抓取、清洗、入库 |
| Base 面板 | 构建渐进式导航的 Obsidian Base 入口面板 |
| 知识变更追踪 | 自动记录操作至 0109-log/，Git commit 追踪文件变更 |
| 定期知识审计 | Claude 定期检查知识库完整性、去重、更新 |

## 三、功能需求详述

### 3.1 目录结构（P0）

严格遵循以下目录树创建所有文件夹：

```
obsidian-vault/
├── 0000-meta/
│   ├── 0000-project-docs/       # 项目文档
│   ├── 0001-templates/          # 笔记模板
│   ├── 0002-scripts/            # Claude Code 脚本
│   ├── 0003-configs/            # 系统配置
│   └── llm-wiki/                # 本项目过程文档
├── 0001-resource/               # 附件资源
├── 0002-example/                # 使用示例
├── 0003-inbox/                  # 知识暂存区
├── 0004-outbox/                 # 知识输出区
├── 0005-markmind/               # 思维导图
├── 0006-excalidraw/             # 绘图
├── 0100-wiki-meta/              # Wiki 元数据
│   ├── INDEX.md                 # 全局元索引（主题目录列表+链接）
│   ├── SCHEMA.md                # 知识结构定义
│   └── RELATIONS.md             # 知识关系图谱
├── 0101-wiki-topics/            # 主题综述（Claude 合成层）
│   └── {大类}/{主题域}.md
├── 0102-wiki-concepts/          # 概念解释（Claude 合成层）
│   └── {大类}/{主题域}.md
├── 0103-wiki-entities/          # 实体档案（Claude 合成层）
│   └── {大类}/{主题域}.md
├── 0104-wiki-comparisons/       # 对比分析（Claude 合成层）
│   └── {大类}/{主题域}.md
├── 0105-wiki-base/              # Base 面板
├── 0109-log/                    # 操作日志（按日拆分）
│   └── LOG-YYYY-MM-DD.md        # 每日日志，追加式
├── 0999-other/                  # 临时工作区
│
├── {编号}-{主题}/               # 个人生活（千位 1-2，按需创建）
│
├── {编号}-{主题}/               # AI 技术（千位 3，按需创建）
│
├── {编号}-{主题}/               # 工作业务（千位 4，按需创建）
│

> 每个主题目录内部结构：
> ```
> {编号}-{主题名}/
> ├── {编号}-{序号}-{slug}.md    # 知识条目
> └── {编号}-{序号}-{slug}.md    # 知识条目
> ```
```

### 3.2 知识 ID 体系（P0）

- **格式**：`{目录编号}-{5位序号}`
- **示例**：`5001-00001`（通用基础知识第1条）、`3001-00042`（AI技术第42条）
- **目录编号规则**：
  - 千位 1-2 → 个人生活（如 `1101-健康`、`1201-投资理财`）
  - 千位 3 → AI 技术（如 `3001-深度学习`、`3002-LLM`）
  - 千位 4 → 工作业务（如 `4001-项目管理`）
  - 千位 5+ → 通用基础（如 `5001-计算机科学`、`5101-物理学`）
  - 主题目录按需创建，不预分配编号

### 3.3 知识 Schema / Frontmatter 标准（P0）

每条知识笔记必须包含以下 frontmatter：

```yaml
---
id: 3001-00001              # 唯一标识（目录编号-序号）
title: "知识标题"            # 标题
topic: "3001-深度学习"       # 所属主题目录名，Claude 自动填写
tags: [tag1, tag2]          # 标签列表，8-10个
aliases: [别名1, slug]      # 别名，含 slug 用于搜索
created: 2026-05-18         # 创建日期
updated: 2026-05-18         # 更新日期
source: "manual|url|file|claude"  # 来源
status: draft|published          # 状态
summary: "一句话摘要"        # 摘要
---
```

### 3.4 知识模板（P0）

需创建以下 Templater 模板：

| 模板 | 用途 |
|------|------|
| `t-concept.md` | 概念类知识（术语定义、抽象概念） |
| `t-entity.md` | 实体类知识（人物、组织、项目） |
| `t-topic.md` | 主题类知识（跨概念综合论述） |
| `t-comparison.md` | 比较类知识（A vs B 对比） |
| `t-inbox.md` | 收件箱原始条目 |
| `t-daily.md` | 日记/日常记录 |

### 3.5 知识摄入（P0 手动 → P1 Claude 辅助 → P2 自动化）

**P0 — 手动摄入**：
- 用户在 inbox 中创建原始笔记
- 手动或通过模板填充元数据
- 手工归档到对应主题目录

**P1 — Claude 辅助加工**：
- 用户在 Claude Code 中指定 inbox 文件
- Claude 读取内容，输出加工后的知识条目
- Claude 建议分类、标签、关联链接

**P2 — 自动化摄入**：
- 用户提供 URL，Claude 自动抓取、清洗、入库
- 用户提供文件（PDF/Word/Markdown），Claude 自动导入
- Claude 对话内容自动沉淀为知识条目

### 3.6 Wiki 元数据（P0 骨架 → P1 内容填充 → P2 自动维护）

**0100-wiki-meta/**：

| 文件 | 内容 | 阶段 |
|------|------|------|
| `INDEX.md` | 全局元索引（手动列出各主题目录链接+统计） | P0 骨架 / P1 内容 |
| `SCHEMA.md` | 知识结构定义（Schema 文档） | P0 |
| `RELATIONS.md` | 知识关系图谱（Canvas 可视化总览） | P0 骨架 / P2 自动 |

**0109-log/**：

| 文件 | 内容 | 阶段 |
|------|------|------|
| `LOG-YYYY-MM-DD.md` | 每日操作日志（追加式，Ingest/Update/Query/Lint 全部操作，含知识变更记录） | P0 骨架 / P1 内容 |

### 3.7 Git 版本控制（P0）

- 全程手动提交，不使用自动提交插件
- 远程仓库：`git@github.com:yu627482453/Mnemosyne.git`
- Claude 操作后直接执行 git add/commit/push
- 提交信息格式：`wiki: {操作} {描述}` 或 `docs: {描述}`

### 3.8 Claude Code 集成（P1）

- **交互模式**：手动触发——用户在 Claude Code 中指定文件/目录
- **工作流**：用户放入 inbox → 在 Claude Code 中 `@` 引用 → Claude 处理 → 输出到目标位置
- **脚本目录**：`0000-meta/0002-scripts/`
- **提示词管理**：`0000-meta/llm-wiki/` 下维护各类处理提示词

### 3.9 标签系统（P3 — 未来计划，本次预留设计）

标签系统将在 MVP 三个阶段全部完成后单独设计，本次不实现，但做以下预留：

- Frontmatter 中 `tags` 字段已定义，支持列表格式
- 目录结构中 `0100-wiki-meta/` 预留 `TAGS.md`（标签索引文件）的位置
- 现有插件 `colored-tags`、`obsidian-tagfolder` 为标签系统提供前端支持
- 设计预期：
  - 层级化标签体系（一级标签/二级标签）
  - 标签与知识 ID 的双向映射
  - 标签自动建议（基于 Claude Code 分析内容）
  - 标签面板的 Base 入口集成

## 四、非功能需求

### 4.1 可扩展性
- 目录结构预留扩展空间（1000-9999 范围内的未分配号段）
- 知识 Schema 支持未来添加新字段
- 模板系统支持新增知识类型

### 4.2 可维护性
- 所有 AI 操作通过 Claude Code 手动触发，可追溯可审核
- Git 记录每次知识变更
- 清晰的文件命名和目录结构便于人工浏览

### 4.3 性能
- 本地 Obsidian vault，无网络依赖
- Dataview 查询需注意大数据量性能
- 附件资源独立存放（0001-resource/）

### 4.4 安全性
- 全本地运行，知识数据不外泄
- Git 本地仓库，不上传远程（当前阶段）
- meld-encrypt 插件支持敏感笔记加密

## 六、约束与假设

### 约束
- 系统完全本地运行，无云同步需求（当前阶段）
- Claude Code 处理为手动触发模式
- Git 仅本地使用

### 假设
- 用户已熟练掌握 Obsidian 基本操作
- Claude Code 已正确安装并配置
- 当前操作系统为 Windows 11

## 七、工作规范

1. **子步骤确认机制**：每个阶段内的每个子步骤在开发前，必须先将该子步骤的设计方案交由用户确认，用户可能会对方案进行调整，确认后方可实施
2. **阶段确认**：每个大阶段完成后暂停，等待用户确认后进入下一阶段
3. **交付物要求**：每个阶段产出 Markdown 文档，保存在 `0000-meta/llm-wiki/` 目录下
4. **问题反馈**：遇到技术难题或需用户决策时，立即停止并说明情况
