# skill-ingest：知识摄入（L1 → L2 → L3）

> 触发：用户 `@` 引用 inbox 文件要求处理

## 前置检查

1. 确认目标文件在 `0003-inbox/` 下且存在
2. 读取全文，提取标题、来源、核心内容

## 拆分判断

满足以下条件拆分为多个 L2 条目：
- 内容有独立标题层级
- 每个概念可独立成文（有定义 + 要点）
- 概念之间有明确边界

不拆分：内容紧密关联、信息量不足独立成文（<3 要点）、用户明确要求合并。

## 执行步骤

### 1. 判断归属主题目录

- 对照 `0000-meta/0003-configs/topics.yaml`
- **首次归档到某主题域，必须询问用户确认**
- 命中已确认映射时允许自动写入
- 边界主题 / 多主题交叉 / 新主题候选必须询问
- 无匹配：按 topics.yaml 规则创建新主题目录

### 2. 生成 slug（D009）

- 英文优先，3-5 推荐选项，`rg --files` 查重，重名追加 `-2`、`-3`

### 3. 创建 L2 标准化主题知识（高保真）

按 `t-knowledge.md` 模板写入：
- L2 是高保真标准化层，默认保留原文主体，不允许过度摘要化
- 默认中文化；英文原文是否保留需用户确认；若保留，放入"原文保留"区
- `tags`：5-10 个，无空格，多词用连字符（如 `multi-agent`）
- `summary`：200-500 字，写独特判断、适用范围、关键机制与限制
- `resource_refs`：本地资源路径列表；正文对应位置用 `![[...]]`
- 来源区：保留作者、机构、原文链接与原始文件位置

### 4. 判断 L3 触发（D004/D013）

L3 由 L2 派生，不得脱离 L2 自建事实。目录按类型组织：

| L3 类型 | 目录组织 | 示例 |
|---------|---------|------|
| concept | 按主题域 | `0102/AI技术/Multi-Agent/orchestrator-worker.md` |
| entity | 按实体类别 | `0103/Products/claude-code.md` |
| comparison | 按比较轴 | `0104/Architecture/single-vs-multi-agent.md` |

entity 类别：Organization, Product, Project, Paper, Person

触发规则：
- entity：独立产品/平台/组织/人物
- concept：机制/方法/原则/模式/功能特性
- comparison：正文主轴是差异/取舍，需指定 comparison_axis
- 同 slug 匹配 → 合并更新（补充、冲突标注、source 追加）
- 主题域首次有内容 → 新建 0101 主题综述

### 5. L3 合并规则

1. 读入已有 L3 页面全文
2. 新信息补充，冲突标注，重复不修改
3. source 字段追加新 L2 路径
4. 更新 updated 日期

### 6. 死链治理

- 正式 `[[wikilink]]` 只链接已存在页面
- 未建概念放入 `planned_links` 或注释 `<!-- TODO: [[可能概念]] -->`
- 坏死链必须修

### 7. 跨主题引用建议

tags 重叠 ≥2 且无 wikilink → 建议关联

### 8. 追加操作日志

```
| HH:MM:SS | Ingest | {路径} | 新增 {标题} |
```

### 9. 报告与收尾

询问是否移入 `.trash/`（D008），用户确认后执行。

## 落盘前验收清单

1. frontmatter 必备字段齐全
2. tags 5-10 个、无空格、多词连字符
3. summary 200-500 字，有独特判断
4. 链接指向正确层级与主题域
5. L3 已明确 source L2 列表
6. resource_refs 与正文 `![[...]]` 对应
7. 涉及新子域或迁移时，旧路径引用已处理
8. 死链已处理

## 字段校验

对照 `schema.yaml` 逐层检查必填字段。
